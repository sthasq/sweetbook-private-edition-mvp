# PlayPick

> 스위트북 바이브코딩 풀스택 개발자 과제 (Book Print API 웹앱 만들기)
> 실제 프로덕트를 만든다는 관점에서 기획, 디자인, 풀스택 구현 및 배포까지 완성했습니다.
> 크리에이터가 인증한 공식 포토북 에디션 + 팬을 위한 AI 채팅 개인화 기능

## 1. 서비스 소개

PlayPick은 크리에이터가 먼저 만든 공식 포토북 에디션을 팬이 AI 채팅 인터뷰로 개인화한 뒤, 실제로 주문 가능한 포토북으로 완성하는 서비스입니다. 
단순한 스크랩북을 넘어, 크리에이터가 승인한 에디션 내에서 팬의 관계와 기억을 녹여 넣는 개인 맞춤형 굿즈 경험을 목표로 했습니다.

**타깃 고객**
- **팬(고객)**: 좋아하는 크리에이터와의 기억을 더 개인적인 방식으로 남기고 싶은 사용자
- **크리에이터(창작자)**: 팬을 위한 공식 포토북 상품을 직접 설계하고 판매하고 싶은 사용자
- **플랫폼 관리자**:  주문, 정산, 인증, 웹훅을 운영 관점에서 관리해야 하는 운영자

**한 줄 가치 제안**
공식 에디션의 신뢰감과 팬 개인화의 감정 밀도를 동시에 가져가는 포토북 플랫폼

---

## 2. 왜 이 서비스를 만들었는가 (개발 과제 목표와 기획 의도)

과제 목표가 "1인 개발자로서 **실제로 서비스할 수 있는 제품**과 **고객 경험**"을 요구했기 때문에, 단순히 API 명세를 따라 기능 몇 개를 붙이는 데모 수준으로는 핵심 목표를 달성할 수 없다고 판단했습니다. 

이에 따라 기능 구현이라는 개발자 지향적인 인상을 넘어, **"고객이 이 서비스에서 왜 시간과 비용을 들여 포토북을 주문해야 하는가?"라는 제품 핵심 가치**와 구체적인 서비스 시나리오(사용자 경험)를 먼저 정의해야만 했습니다. 

기존 팬 굿즈는 대개 정적이고, 반대로 완전히 개인 제작형 포토북은 공식 상품 느낌이 약해지기 쉽습니다. PlayPick은 이 사이를 연결합니다.
1. 크리에이터가 먼저 공식 에디션을 발행한다.
2. 팬은 그 안에서 AI 인터뷰를 통해 자기 이야기와 감정을 넣는다.
3. 결과물은 단순 업로드 앨범이 아니라 공식 굿즈처럼 느껴지는 개인화 포토북이 된다.

## 3. 핵심 기능

### 3-1. 팬(소비자) 이용 흐름
- 공개된 에디션 목록 탐색 및 상세 확인
- AI 채팅 인터뷰로 개인화 정보 수집 (제안된 답변 `suggestedReplies` 지원)
- 포토북 프리뷰 확인
- 인쇄용 포토북 생성 / 확정
- 배송 정보 입력 및 제작/배송 원가 + 서비스 금액 확인
- 토스 페이먼츠(테스트) 기반 결제 진행 및 주문 완료
- 나의 포토북에서 통합 관리

### 3-2. 크리에이터 이용 흐름
- 스튜디오에서 에디션 생성 / 수정 / 발행 (단계형 편집 흐름)
- 큐레이션 이미지 업로드
- 발행 후 팬 시점 미리보기 / 공유 링크 확인
- 스튜디오에서 발생한 주문 조회 가능

### 3-3. 관리자 이용 흐름
- 관리자 대시보드 (주문 조회, 정산, 사용자/크리에이터 권한 관리)
- 웹훅 로그 조회 및 SSE 기반 실시간 웹훅 모니터링
- 주요 엔티티 페이지네이션 및 상태 추적

### 3-4. 제품 및 기술 주요 차별점
- 세션 기반 인증 + CSRF 방어, `기본 접근 차단(Deny-by-default)` 접근 제어
- Sweetbook 웹훅 HMAC 서명 + timestamp 검증 + `delivery_uid` 기반 중복 처리 방지(Deduplication)
- Redis 기반 세션 저장 및 백엔드 캐시
- 외부 API 실패 시 롤백 및 재시도를 보장하는 안전한 주문 트랜잭션 흐름

## 4. 설계 의도 및 중요하게 본 판단

### 4-1. 선택과 집중: 화려한 데모보다 선명한 '제품 경험'
개발 초기에는 YouTube/OAuth 연동, AI 이미지 생성 등 다양한 기술적 요소를 포함했지만, 기능이 늘수록 핵심이 흐려진다는 판단 하에, 설명 비용이 큰 요소를 과감히 제거했습니다. 결과적으로 기능의 가짓수를 줄이는 대신, **팬이 인플루언서 에디션을 AI 채팅으로 개인화하고 미리보기를 거쳐 실제로 주문(E2E)하는 단일 흐름**에 집중하여 핵심 메시지를 훨씬 간결하고 설득력 있게 만들 수 있었습니다.

### 4-2. 공식 에디션에 더해진 팬 개인화
사용자가 처음부터 백지에서 책을 만드는 서비스가 아니라, 크리에이터가 승인한 공식 에디션 위에 팬이 개인화 과정만을 얹는 구조로 기획했습니다. 이 제약 덕분에 산출물이 단순 유저 생성 콘텐츠가 아니라, 실제 판매 가능한 공식 크리에이터 굿즈(팬 굿즈)처럼 보이도록 설계할 수 있었습니다.

### 4-3. 폼 입력 대신 AI 채팅 인터뷰
초기 입력 폼/외부 연동 중심 흐름 대신, 최종적으로는 **AI 채팅 인터뷰**를 메인 개인화 경험으로 재정의했습니다. 사용자가 더 가볍게 시작할 수 있고, 서비스 톤에 맞는 자연스러운 입력 경험을 제공하며, 최종 포토북 문구(`bookCopy`) 생성까지 자연스럽게 연결됩니다.

### 4-4. 배송/결제 단계의 책임 분리
초기 UI는 생성, 최종화, 견적, 결제가 한 화면에 몰려 있었지만, 최종 흐름에서는 **미리보기 단계에서 포토북 생성/확정**, **배송 단계에서 배송 정보 입력/견적/결제**로 역할을 엄격히 분리했습니다. 이로 인해 트랜잭션의 책임이 분리되고, 팬 입장에서 “지금 무엇을 해야 하는가”가 훨씬 분명해졌습니다.

### 4-5. 프리뷰는 웹 페이지가 아니라 “책처럼” 보여야 한다
프리뷰 화면은 단순한 API 응답 데이터 나열이 아닌, **표지 단독 흐름, 내지 24p 전개, 마지막 발행면 표출, 책장 넘김 애니메이션 구조(`react-pageflip` 활용)**로 재정리하여 완성도 있는 경험을 제공했습니다.

## 6. 로컬 실행 방법

권장 실행 방식은 **Docker Compose** 기준입니다.

```bash
# 1. 환경변수 설정
cp .env.example .env

# 2. 통합 환경(React 19 + Spring Boot 3.5 + MySQL 8 + Redis 7) 백그라운드 띄우기
docker compose up -d --build
```
- **Frontend** (팬 UI, 스튜디오, 운영 콘솔 등 데모 홈): `http://localhost:3000`
- **Backend API** (Swagger 등 포함): `http://localhost:8080/swagger-ui/index.html`

*기본적인 환경에서는 미리 구성된 시나리오와 더미 응답으로 전체 흐름 체험이 가능하며, API의 완벽한 실연동 확정을 위해서는 `.env`에 `SWEETBOOK_API_KEY` 및 `OPENROUTER_API_KEY`, `TOSS_PAYMENTS_CLIENT_KEY` 입력이 필요합니다.*

## 8. Book Print API 사용 내역 및 연동 회고

Book Print API의 전체 제작 흐름과 PlayPick 내 활용 내역은 아래와 같습니다.

| API 용도 | Book Print API 엔드포인트 | PlayPick 활용 |
| --- | --- | --- |
| **템플릿 상세 레이아웃 탐색** | `GET /templates/{templateUid}` | 프론트엔드 미리보기를 위해 에디션의 원본 페이지 구조 파악 |
| **신규 포토북 세션 활성화** | `POST /books` | AI 개인화 완료 후 임시 저장 기반을 둘 빈 책(`bookUid`) 발급 |
| **표지 그래픽 정보 반영** | `POST /books/{uid}/cover` | AI가 수집한 텍스트를 표지 구성 스펙에 맞춰 업데이트 |
| **내지 구성 및 구조 반영** | `POST /books/{uid}/contents` | 내부 페이지별 사진 및 큐레이션 자산, 레이아웃 조합 업데이트 |
| **최종 인쇄 확정 픽업** | `POST /books/{uid}/finalization` | 팬이 '인쇄 확정' 클릭 시 편집 상태 잠금 (인쇄 준비) |
| **배송비 추정 / 결제 대기** | `POST /orders/estimate` | 결제에 넘길 정확한 제작/배송비 및 결제 원금 파악 |
| **실 주문 제작 발주** | `POST /orders` | PG 결제 승인 확인 직후 안전하게 발주 처리 |
| **웹훅 이벤트 트래킹** | `POST /api/sweetbook/webhook` | 제작 상태 변경 이벤트를 수신 (Webhook HMAC 검증 및 대시보드 SSE) |

## 9. 가격 모델과 운영 모델

PlayPick은 단순 결제액 배분(Revenue Share) 구조에서 탈피하여, 실제 주문 시점을 스냅샷 삼아 **Sweetbook 원가(`vendor_cost`) + 플랫폼 서비스 마진(`margin`) 단위로 엄격히 분리하여 정산 구조를 설계**했습니다.
주문 시점에는 다음 값을 스냅샷(Snapshot)으로 저장합니다: `vendor_cost`, `margin_rate`, `margin_amount`, `platform_fee`, `creator_payout`.
이 아키텍처를 선택한 이유는 플랫폼에서 실원가를 투명하게 분리해 수익을 명확히 하고, 향후 가격 분배 정책이 바뀌어도(예: 크리에이터별 차등 Premium 등) 기수행 주문들의 정산 무결성을 변함없이 보존할 수 있기 때문입니다.

## 10. AI 활용 및 트러블슈팅 경험

이 프로젝트는 초기 기획부터 최종 코드 생성, 카피라이팅까지 전방위적으로 AI 도구를 활용하여 빠른 사이클을 달성했습니다. 전체 개발 경험은 **“AI로 빠르게 가설을 검증하고, 직접 결과를 확인하며 불필요한 것을 걷어내 선명한 프로덕트로 수렴해가는 여정”**에 가까웠습니다.

### 10-1. 서비스 기동에 사용한 AI (OpenRouter 기반 LLM)
크리에이터 페르소나와의 1:1 대화 몰입감을 위한 팬 스튜디오 AI 채팅, 사용자가 바로 누를 수 있는 인터랙티브 추천 답변(`suggestedReplies`), 최종 포토북 문구(`bookCopy`) 생성에 활용되었습니다.

**🔥 트러블슈팅: LLM JSON 폴백 로직 구축**
서비스 내 LLM 구동 시 프롬프트를 벗어나 비구조적 텍스트나 잘못된 형식을 반환하여 전체 개인화 흐름과 리액트 화면이 붕괴되는 현상이 있었습니다. 이를 해결하기 위해 프롬프트의 제약력만 믿지 않고, **JSON 파싱 실패가 일어날 시 정규식 기반으로 원본 텍스트를 파싱하거나 우회적으로 안전한 디폴트 응답을 반환하는 견고한 시스템 레벨 폴백 로직(Fallback Logic)**을 추가했습니다.

### 10-2. 소프트웨어 엔지니어링 과정에 사용한 AI 도구
- **Codex 메인 활용 및 다중 모델 교차 검증**: 프론트-백엔드 보일러플레이트, 인증 상태, 서버 로직 등 전반적인 코드 생성은 주로 **Codex**를 메인으로 사용했습니다. 반면, **UI/UX 디자인 및 세밀한 스타일링 작업에서는 Claude, Gemini 등 디자인 추론에 강한 모델을 Cursor, 안티그래비티, VS Code Copilot 등을 통해 교차 활용하며 품질을 높였습니다.**

**🔥 작업 회고: 모호함의 증폭과 지식베이스를 이용한 방어**
에이전트가 코드를 대신 짜주는 편의성은 압도적이었으나, 반대로 기획 의도가 조금이라도 모호해지면 그 불명확함이 코드 전반으로 빠르게 증폭된다는 점을 일찍 인지했습니다. 
이를 예방하기 위해 마크다운을 통해 작업 회고 및 컨텍스트를 꾸준히 남김으로써 AI에게 지속적으로 목표 컨텍스트를 공급했고, 세션별로 **"이번엔 무엇을 남기고, 무엇을 버릴 것인가"를 확실히 정리하는 프롬프팅 습관**을 통해 방향성을 잃지 않았습니다.

## 11. 시스템 한계 및 확장 관점 (더 시간이 있었다면)

- **AI 생성 이미지 기반의 팬덤 경험 극대화**: 기획 초기에는 인플루언서와 팬의 이미지를 활용해 둘이 함께 있는 듯한 AI 생성 합성 사진을 포토북 내지에 포함하는 기능을 구상했습니다. 이를 도입하면 텍스트 중심의 대화를 넘어 시각적으로 몰입감이 극대화된 진정한 팬덤 경험으로 확장할 수 있을 것입니다.
- **웹훅 이벤트 기반의 서비스 이용자(Fan) 알림 시스템 연동**: 현재는 Book Print API의 웹훅 이벤트를 관리자 대시보드(SSE)와 주문 상태 DB 업데이트 목적으로만 소비하고 있습니다. 프로덕션 환경이라면 이 웹훅 데이터를 기반으로 주문자에게 '제작 시작', '배송 출발' 등의 이메일/알림톡을 비동기적으로 발송하는 **이벤트 기반 알림 파이프라인으로 확장하는 것이 좋아보입니다.**
