# 2026-04-10 작업 회고

## 이번에 정리한 것

- 팬 전용이던 회원가입 흐름에 크리에이터 가입 경로를 추가했다.
- Studio에서 내부 에디션 템플릿과 Sweetbook 레이아웃 템플릿을 분리해서 보이도록 정리했다.
- 큐레이션 자산과 Sweetbook 레이아웃 모두 시각적인 프리뷰를 넣어 선택 판단을 빠르게 만들었다.
- 로컬 백엔드 기본 실행을 Docker MySQL 기준으로 맞추고, 실행 중 흔한 충돌 원인을 더 빨리 드러내도록 정리했다.
- Studio를 단계형 플로우로 재구성하고, 템플릿 선택 구간을 접기/펼치기와 가로 스와이프 중심으로 다시 정리했다.
- YouTube 리캡 자산 불러오기를 추가하고, 채널 링크·핸들·인코딩 URL을 더 안정적으로 해석하도록 보강했다.

## 잘된 점

- 사용자 피드백이 바로 UI/용어 변경으로 이어졌다. `채널 핸들`처럼 낯선 표현을 `크리에이터 아이디`로 빠르게 바꾼 흐름이 좋았다.
- Studio를 단순 기능 추가가 아니라 정보 구조 관점에서 다시 나눠서, 내부 템플릿과 출력 레이아웃의 역할 차이를 명확히 만들 수 있었다.
- Playwright로 실제 브라우저 상태를 함께 보면서 작업해 UI 깨짐과 흐름 문제를 텍스트 추측 없이 바로 확인할 수 있었다.
- 백엔드, 프론트엔드, 실행 환경 문제를 한 번에 엮어 해결해서 기능이 코드상 구현만 되고 실제로는 안 도는 상태를 피했다.
- 템플릿 단계에서 '너무 길고 많다'는 피드백을 받아, 기능 추가보다 먼저 정보량을 줄이고 스캔하기 쉽게 재배치한 판단이 유효했다.
- YouTube 리캡 기능을 붙인 뒤 바로 실제 채널 링크로 검증하면서, 해석 정확도와 운영 중 에러를 빠르게 잡아낼 수 있었다.

## 아쉬웠던 점

- MySQL 전환 과정에서 Docker 컨테이너, 로컬 MySQL 서비스, `.env` 비밀번호가 한동안 서로 엇갈려 문제 진단이 길어졌다.
- StudioPage 변경 폭이 커지면서 화면 구조, 템플릿 선택, 프리뷰 로직이 한 파일에 많이 몰렸다.
- Sweetbook 템플릿 프리뷰는 실제 썸네일 API가 아니라 이름 패턴 기반 목업이라 정확도에 한계가 있다.
- YouTube 링크 해석은 생각보다 입력 변형이 많아서, `@핸들`, 한글 핸들, 퍼센트 인코딩 URL, 검색 fallback을 구분하지 않으면 오인식하기 쉬웠다.
- 실행 스크립트가 있어도 기존 셸 환경 변수나 오래된 Docker 볼륨이 섞이면 다시 `root` 계정으로 붙는 문제가 재발할 수 있었다.

## 다음에 개선할 점

- Sweetbook 템플릿 메타데이터에 실제 미리보기 이미지나 설명이 있다면 API 응답까지 연결해 목업 프리뷰를 대체한다.
- StudioPage를 템플릿 선택, 자산 편집, 요약 패널 단위로 컴포넌트 분리해서 유지보수 부담을 줄인다.
- 로컬 실행 스크립트에 Docker 상태, 포트 점유, 인증 정보 검증을 조금 더 일찍 안내해 초기 진입 비용을 낮춘다.
- 크리에이터 가입 이후 승인/검수 흐름이 필요한지 제품 정책 기준으로 다시 정리한다.
- YouTube 리캡 불러오기 전에 "해석된 채널"을 먼저 미리 보여줘서, 다른 채널로 인식됐는지 사용자가 클릭 전에 확인할 수 있게 한다.
- 큰 응답을 받는 YouTube API는 요청 단위와 버퍼 한도를 함께 관리해서, `DataBufferLimitException` 같은 운영성 문제를 미리 줄인다.

## 남은 체크포인트

- 선택한 Sweetbook 레이아웃이 실제 출판 결과물에서 기대한 형태로 반영되는지 end-to-end 확인이 더 필요하다.
- 회고 시점 기준으로는 기능이 로컬에서 동작하지만, 배포 환경의 시크릿과 DB 설정은 별도 확인이 필요하다.
- 인코딩된 한글 핸들 URL과 실제 브라우저 입력 흐름에서 리캡 불러오기가 완전히 동일하게 동작하는지 한 번 더 E2E로 점검할 필요가 있다.
