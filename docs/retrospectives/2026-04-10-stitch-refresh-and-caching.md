# 2026-04-10 Stitch Refresh / Caching 회고

## 이번 배치에서 한 일

- Stitch 산출물을 정리해 주요 팬/크리에이터 화면의 디자인 방향과 HTML 목업, 스크린샷을 남겼다.
- 실제 프런트 구현에서 페이지 전환이 너무 급하게 느껴지던 부분을 페이드 중심으로 완화했다.
- 전체 화면 로딩 문구가 짧은 전환마다 튀어나오는 문제를 줄이기 위해, 스피너를 지연 노출 방식으로 바꿨다.
- 백엔드에 Redis 기반 세션 저장소를 도입해, 다중 인스턴스나 재시작 상황에서도 로그인 세션을 안정적으로 유지할 기반을 만들었다.
- Sweetbook `book-specs`, `templates` 조회를 Redis 공유 캐시로 바꿔, 인스턴스별 메모리 캐시 한계를 줄였다.
- 프런트 API 클라이언트에 메모리 캐시와 in-flight dedupe를 추가해, 같은 GET 요청을 짧은 시간 안에 반복 호출하지 않도록 했다.
- 프로젝트/에디션/인증 관련 mutation 이후 필요한 경로만 invalidation 하도록 맞춰 stale 데이터를 줄였다.
- Docker Compose, 로컬 실행 스크립트, README, `.env.example`를 함께 갱신해 실행 경로와 운영 구성을 문서화했다.

## 잘된 점

- 처음엔 라우트 전환을 복잡하게 제어하려다 멈춤이 생겼지만, 빠르게 방향을 바꿔 더 단순하고 안전한 해법으로 수습했다.
- Redis 도입을 세션 저장소에서 끝내지 않고, Sweetbook 조회 캐시까지 연결해 운영 관점의 효용을 분명하게 만들었다.
- 테스트가 Redis 없이도 깨지지 않도록 `session-redis` 프로필로 기능을 분리해, 개발/테스트/운영 경계를 비교적 깔끔하게 유지했다.
- 프런트 캐시는 라이브러리를 크게 들이지 않고 지금 구조에 맞는 최소 구현으로 넣어, 도입 비용 대비 체감 개선 가능성을 확보했다.
- API mutation 이후 관련 캐시를 명시적으로 비우도록 해서, 단순 캐시 추가로 인한 stale 데이터 리스크를 어느 정도 통제했다.
- 문서, 스크립트, docker 설정까지 같이 업데이트해 “코드만 되고 실행은 안 되는” 상태를 피했다.

## 아쉬웠던 점

- 페이지 전환 UX를 개선하는 과정에서 preload 방식이 한 번 실패했고, 그 때문에 구현을 되돌리는 비용이 들었다.
- 현재 브랜치 범위가 꽤 넓다. 디자인 산출물, UX 조정, Redis 인프라, 프런트 캐시가 한 PR에 같이 들어가 리뷰 포인트가 많아졌다.
- 프런트 캐시는 지금은 수동 TTL + 수동 invalidation 기반이라, 데이터 종류가 더 늘어나면 관리 포인트가 금방 많아질 수 있다.
- Redis 세션은 잘 분리했지만, 캐시 TTL과 eviction 정책은 아직 제품 usage 패턴을 충분히 본 뒤 더 다듬어야 한다.
- Stitch 산출물과 실제 구현 코드 사이에 아직 자동 연결은 없어서, 디자인 산출물이 참조 자료 역할에 머무른다.

## 다음에 개선할 점

- 프런트 캐시가 더 커지기 전에 TanStack Query 같은 도구 도입 시점을 검토해, invalidation과 stale 정책을 체계화한다.
- 공개 에디션 목록과 상세도 서버/프런트 캐시를 함께 보는 구조로 정리해, 첫 진입 체감까지 더 줄인다.
- PR 범위를 더 잘게 나눌 수 있다면 디자인 산출물과 인프라/캐시 변경은 분리해 리뷰 부담을 낮춘다.
- Redis 캐시 적중률이나 세션 유지 효과를 실제 운영 로그로 관찰할 수 있게 metrics를 붙인다.
- 페이지 전환은 현재 페이드 중심으로 안정화했지만, 다음엔 링크 hover prefetch까지 붙여 첫 진입 로딩도 함께 줄인다.

## 남은 체크포인트

- Docker 기반 `session-redis` 실행에서 실제 로그인 세션이 백엔드 재시작 후에도 유지되는지 수동 검증이 한 번 더 필요하다.
- Sweetbook 템플릿/규격 캐시가 TTL 이후 정상 갱신되는지 운영 로그 기준으로 확인할 필요가 있다.
- 프런트 캐시가 로그인 전후, 주문 직후, 퍼블리시 직후 경로에서 stale 화면을 남기지 않는지 브라우저 기준 E2E 확인이 필요하다.
- 현재 PR 범위에 포함된 Stitch 산출물이 정말 저장소에 남겨야 하는 산출물인지, 장기적으로는 output 정리 정책이 필요하다.
