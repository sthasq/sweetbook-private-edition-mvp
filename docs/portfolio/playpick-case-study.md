# PlayPick Case Study

## 1. 프로젝트 한 줄 설명

PlayPick은 Sweetbook Book Print API를 활용해, 크리에이터가 만든 공식 에디션을 팬이 AI 인터뷰로 개인화하고 실제 주문까지 이어갈 수 있게 만든 포토북 커머스 프로젝트입니다.

README가 저장소 첫 화면에서 빠르게 훑는 문서라면, 이 문서는 그 뒤에서 "왜 이렇게 만들었는지"를 더 자세히 설명하는 용도입니다.

---

## 2. 왜 이 프로젝트를 만들었나

이 프로젝트를 시작할 때 가장 먼저 든 생각은, 단순히 외부 API를 붙이는 것만으로는 제품처럼 보이기 어렵다는 점이었습니다.

Sweetbook API는 책을 만들고 주문하는 데 필요한 기능을 제공하지만, 사용자가 실제로 느끼는 가치는 그 위에서 어떤 흐름을 설계하느냐에 따라 달라집니다. 그래서 저는 이 프로젝트를 단순한 API 데모가 아니라, 다음 질문을 풀어보는 작업으로 접근했습니다.

- 팬은 왜 이 서비스에서 시간을 들여 개인화를 해야 할까
- 크리에이터는 왜 이 플랫폼에서 공식 에디션을 만들고 싶어질까
- 운영자는 주문과 웹훅 상태를 어떻게 확인해야 할까

결과적으로 "공식 굿즈의 신뢰감"과 "개인화 경험의 감정 밀도"를 동시에 가져가는 흐름을 만드는 것이 핵심이라고 판단했습니다.

---

## 3. 문제를 어떻게 정의했나

기존 팬 굿즈는 대체로 정적이고, 사용자가 자신의 이야기를 넣을 여지가 많지 않습니다. 반대로 완전히 사용자가 직접 만드는 포토북은 자유롭지만, 공식 상품처럼 느껴지는 힘이 약합니다.

PlayPick은 이 중간 지점을 노렸습니다.

1. 크리에이터가 먼저 공식 에디션을 발행한다.
2. 팬은 AI 채팅 인터뷰를 통해 추억과 감정을 입력한다.
3. 결과물은 개인화가 반영되지만, 여전히 공식 굿즈처럼 보이는 책으로 완성된다.

처음에는 더 많은 기능을 넣는 방향도 고민했지만, 기능이 많아질수록 제품 메시지가 흐려질 가능성이 컸습니다. 그래서 결과적으로는 하나의 핵심 흐름을 끝까지 연결하는 데 집중했습니다.

---

## 4. 핵심 사용자 흐름

### 팬

- 에디션 탐색
- AI 인터뷰 개인화
- 포토북 프리뷰
- 책 생성 및 확정
- 배송 정보 입력
- 견적 계산
- 결제 또는 데모 주문

### 크리에이터

- 에디션 초안 작성
- 표지와 큐레이션 자산 구성
- 에디션 발행
- 팬 시점 미리보기 확인
- 주문 현황 확인

### 관리자

- 대시보드에서 주문과 정산 확인
- 전체 주문/사용자 조회
- Sweetbook 웹훅 로그 확인
- SSE 기반 실시간 웹훅 스트림 모니터링

이 세 역할을 함께 두었기 때문에, 단순한 팬용 프론트엔드 프로젝트가 아니라 하나의 서비스처럼 보이도록 만들 수 있었습니다.

## 5. 주요 설계 선택

### 5-1. 백지 제작보다 공식 에디션 중심 구조

사용자가 처음부터 모든 것을 만드는 방식은 자유롭지만, 공식 굿즈 같은 느낌을 주기 어렵다고 생각했습니다. 그래서 크리에이터가 먼저 에디션을 설계하고, 팬은 그 안에서 개인화만 더하는 구조를 선택했습니다.

이 방식의 장점은 다음과 같았습니다.

- 결과물이 더 일관된 톤을 유지한다
- 크리에이터 역할이 분명해진다
- 팬 입장에서 시작 장벽이 낮아진다

### 5-2. 폼 입력보다 AI 채팅 인터뷰

개인화는 단순 정보 입력이 아니라 감정 이입 과정이라고 생각했습니다. 그래서 긴 폼보다는 대화형 흐름이 더 잘 맞는다고 판단했습니다.

이 선택 덕분에 다음을 기대할 수 있었습니다.

- 답변 예시를 통해 시작 장벽을 낮출 수 있다
- 사용자가 더 자연스럽게 자신의 이야기를 풀어낼 수 있다
- 최종 포토북 문구 생성까지 흐름이 이어진다

### 5-3. 한 번에 처리하지 않고 단계 분리

책 생성, 확정, 가격 계산, 결제/주문을 한 화면에 몰아두면 사용자는 지금 무엇을 해야 하는지 헷갈리고, 서버 책임도 뒤섞이기 쉽습니다. 그래서 이 흐름을 단계별 API로 나눴습니다.

```text
generate-book -> finalize-book -> estimate -> payment-session / order
```

이 구조는 UX뿐 아니라 디버깅 관점에서도 도움이 됐습니다. 어느 단계에서 실패했는지 비교적 분명하게 확인할 수 있었기 때문입니다.

## 6. 구현하면서 중요했던 백엔드 포인트

### 세션 인증과 접근 제어

로그인 이후에는 세션 기반 인증을 사용했고, CSRF 보호와 역할 기반 권한 제어를 함께 적용했습니다. 공개되어야 하는 엔드포인트와 보호되어야 하는 엔드포인트를 분리하면서, 무심코 열어두는 경로가 없도록 기본 정책을 보수적으로 가져가려고 했습니다.

### Sweetbook 웹훅 처리

웹훅은 단순히 받는 것보다 "안전하게 받을 것"과 "운영자가 볼 수 있을 것"이 더 중요하다고 생각했습니다.

그래서 다음 요소를 함께 구현했습니다.

- `timestamp + rawBody` 기반 HMAC 검증
- 허용 오차 범위 기반 timestamp 확인
- `delivery_uid` 기반 중복 이벤트 무해화
- 관리자 화면 SSE 스트림 반영

이 과정에서 웹훅은 단순 부가 기능이 아니라 운영 기능이라는 점을 많이 배웠습니다.

### 가격 스냅샷

주문 시점에 `vendor_cost`, `margin`, `platform_fee`, `creator_payout`을 저장하도록 한 것은, 나중에 정책이 바뀌더라도 과거 주문의 기준이 흔들리지 않게 하기 위해서였습니다. 작은 프로젝트라도 정산 기준은 나중에 꼬이기 쉬운 부분이라 미리 구조를 잡아두고 싶었습니다.

## 7. 프론트엔드에서 신경 쓴 부분

### 역할별 화면 분리

팬, 크리에이터, 관리자 화면을 완전히 같은 맥락으로 두기보다, 역할별로 해야 하는 일이 다르다는 점이 잘 드러나도록 나누었습니다.

### 포토북 프리뷰 경험

프리뷰는 단순 JSON 결과를 보여주는 화면이 아니라, 실제 책을 넘기는 듯한 감각을 주고 싶었습니다. 그래서 `react-pageflip`을 사용해 책처럼 넘기며 확인하는 형태로 만들었습니다.

### 관리자 실시간 반응

웹훅 이벤트가 들어왔을 때 운영자가 바로 알 수 있도록 SSE 기반 실시간 반영을 붙였습니다. 이 덕분에 관리자 화면도 단순 조회 화면이 아니라 살아 있는 운영 화면처럼 보이게 만들 수 있었습니다.

## 8. 포트폴리오로서 중요했던 판단

### 외부 API 키 없이도 보여줄 수 있어야 한다

포트폴리오 프로젝트는 보는 사람이 모든 API 키를 갖고 있지 않아도 흐름을 확인할 수 있어야 합니다. 그래서 실연동만 고집하지 않고 시뮬레이션 모드를 함께 유지했습니다.

- `OPENROUTER_API_KEY` 없음 -> 결정적 응답 기반 AI 인터뷰
- `SWEETBOOK_API_KEY` 없음 -> 책 생성, 견적, 주문 시뮬레이션
- `TOSS_PAYMENTS_*` 없음 -> 데모 주문 흐름으로 전환

이 구조 덕분에 로컬 개발과 공개 데모 양쪽에서 확인하기 편했습니다.

### README와 케이스 스터디를 분리

README에는 첫 화면에서 바로 읽혀야 하는 내용만 두고, 더 깊은 설명은 별도 문서로 나누는 편이 좋다고 판단했습니다. README가 너무 길어지면 오히려 중요한 내용이 안 보일 수 있기 때문입니다.

## 9. 아쉬웠던 점

- 더 많은 수치를 남기지 못했습니다.
- 운영 관측성이나 로그 체계를 더 정리하면 좋았을 것 같습니다.
- 크리에이터 스튜디오 경험도 더 세밀하게 다듬을 여지가 남아 있습니다.
- 웹훅 이후 주문자 알림까지 이어지는 비동기 파이프라인은 아직 다음 단계로 남아 있습니다.

---

## 10. 이 프로젝트에서 배운 점

- 외부 API 연동 프로젝트에서도 결국 중요한 것은 사용자 흐름이라는 점
- 보안과 운영은 뒤로 미루면 나중에 더 어려워진다는 점
- README도 설치 문서가 아니라 하나의 제품 소개 문서가 될 수 있다는 점
- 기능을 많이 넣는 것보다 핵심 흐름을 끝까지 연결하는 것이 더 설득력 있다는 점
